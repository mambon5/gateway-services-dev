class mqtt_interaction(subtopic=None, pubtopic=None, mode='request', file_settings='cloud_settings.py', interlocutor='cloud', wni=None, need_local_wni=True)[source]

Bases: object

Basic interaction class. The classes mqtt_interaction_cloud and mqtt_interaction_node_basic are extensions of this one.

The structure of this class is the following:

Structure

  1. It creates the WNI if needed, and not already created.

  2. If the interlocutor is the “cloud”, then it subscribe and sets a publisher to the right global MQTT topics.

  3. Finally, it has the following methods:

    • listen()

    • on_message()

    • send_message()

    • start_listen_thread()

    • check_for_response()

    • on_disconnect()

    • on_publish()

    • update_with_new_message()

    • update_msg_id()

    • set_node_id()

    • stop_sub_loop()

    • print_sub_loop()

    • start_listen_loop()

    • send_msg_from_input()

    • is_a_msg_request()

    • convert_dst_address()

Parameters

  • subtopic (string) – MQTT topic to which the client will subscribe to

  • pubtopic (string) – MQTT topic to which the client will publish to

  • mode (string) – mode of the interaction. Can be either “listen” to receive messages from the subtopic, or “request” if we want to the entity to publish requests to the pubtopic

  • file_settings (file) – file containing the shared message id to update

  • interlocutor (string) – entity which we wish to communicate to. Defaults to cloud

Returns

Returns a mqtt_interaction class with the following attributes:

check_for_response(msg)[source]

Checks whether the message received corresponds to any previous request.

Does

  1. Checks weather the response msg id matches the request msg id.

  2. If the above is true, and either the node ids or the gw ids of the received and sent messages coincide, then the received message is considered a received response to a previous request.

  3. If that is the case, the algorithm marks it as a received response, and stops the listening thread by calling the stop_sub_loop() function.

  4. If the received message is not a response, then that is reported in the logs/console print.

Parameters

msg (RxMsg) – message class

Returns

True

– if response id matches the initial request

False

– Otherwise

publish(payload, gw_id=None, sink_id=None, node_id=None, client=None)[source]

Publishes a message to the MQTT topic.

Does

  1. Starts a listening thread to anticipate the possible response. It calls the function start_listen_thread().

  2. It calls the publishing function, which can either be:

    • publish_c() if we are publishing the message to the cloud

    or

    • publish_meshnet() if we are publishing the message to the nodes.

  3. Checks if the message has been published successfully or not, and reports a message accordingly to the logs.

  4. Starts the timeout countdown for receiving a response to the just published request message. This is done by setting boolean start_countdown to True.

Parameters

  • client (mqtt.Client) – publisher client

  • payload (bytes) – message payload

  • id (node) – node id to who we wish to communicate

  • gw_id (int) – gateway id to send the message from (in case of fake cloud, to)

  • sink_id (string) – Sink id where nodes connect to the gateway

Returns

  • published (bool) – True if publish successful, False if failed.

start_listen_loop()[source]

Creates and starts the loop that prints the timeout time left before aborting the connection to either the cloud or the nodes.

Does
If interaction mode == “listen”

  1. It does nothing, either the cloud or the node class is responsible for properly listening to all the incoming messages.

If mode == “request”

  1. Creates a loop that lasts for max_wait time at most. This is a parameter of the RxMsg class.

  2. Once a matching response to the previous request, or the max_wait time is reached, the loop stops.

start_listen_thread()[source]

stop_sub_loop()[source]

Stops the listening loop that actually keeps printing the remaning time before timeout, when a message is sent to either the cloud or the nodes.

Does

  1. sets the variable *run_loop” to False

  2. If the interlocutor is cloud it calls the paho mqtt function loop_stop() to stop the subscription to the global MQTT broker.
