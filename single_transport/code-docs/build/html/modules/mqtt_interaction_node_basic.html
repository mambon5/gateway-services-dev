Basic mesh network mqtt interaction class.

The classes mqtt_interaction_node_main() and mqtt_interaction_node_temporal() are extensions of this one.

The structure of this class is the following:

Structure
  1. It initializes the sub class mqtt_interaction() and gives the following parameters to it:

    • subtopic

    • pubtopic

    • mode

    • local file for message id upadtes

    • interlocutor

    • WNI (can be None)

  2. Sets the timout time for listening to a response message, to 7 seconds.

  3. Finally, it has the following methods:

    • on_message_nodes()

    • publish_meshnet()

    • listen_to_nodes()

    • send_message_to_nodes_from_input()

    • request_to_node()

    • on_gateway_answer_callback()

Parameters

  • subtopic (string) – MQTT topic to which the client will subscribe to

  • pubtopic (string) – MQTT topic to which the client will publish to

  • mode (string) – mode of the interaction. Can be either “listen” to receive messages from the subtopic, or “request” if we want to the entity to publish requests to the pubtopic

  • file_settings (file) – file containing the shared message id to update

  • interlocutor (string) – entity which we wish to communicate to. Defaults to cloud

Returns

Returns a mqtt_interaction_node_basic class with the following attributes:

pubclient

(mqtt.Client) – MQTT client for publishing messages (only if interlocutor is “cloud”)

subclient

(mqtt.Client) – MQTT client for subscribing to a topic (only if interlocutor is “cloud”)

pub_msg_id

(int) – unique message temporal identifier for the published messages

gw_id

(int) – gateway id for the interaction

run_loop

(bool) – run the subclient loop, while true

pub_resp_received

(bool) – whether a response for the sent message has been received

max_wait

(int) – max waiting time for response before deciding no response has been received, in seconds

startime

(int) – start time (in seconds) of the loop in the subclient

thread

(thread) – thread to use for starting a subscriber or publisher client when needed while the main thread still runs

print_raw_data

(bool) – whether or not we wish to print the raw data we get from the node

tunnel

(cloud_to_node_tunnel) – class for establishing a connection between a cloud interaction and a node interaction

node_response

(RxMsg) – The message containing the received response from the nodes

wni

(WirepasNetworkInterface) – a wirepas network interface class connected to the selected local mqtt broker

listen_to_nodes(prefix)[source]

Listens to node messages.

Does

  1. Sets a wirepass filter

  2. Sets a callback on any received data from the nodes, in our cass the cb is on_message()

  3. If the mode is “request”, it will:

    • start the start_listen_loop() loop, to continuously listen to wirepass messages

    • Once the loop finishes (for whatever reason) it removes the wirepas filter for the callback

    If the mode is “listen” it doesn’t do anything else.

Note

This is the equivalent function to the listen_to_cloud() function in the interaction cloud class.

Parameters

prefix (string) – parent prefix to append to the current prefix. This will be reported in all prints made by this function

Returns

  • True if successful, an error if not.

on_message_nodes(data)[source]

Every time a message is received from the mesh network, this function is invoked.

Does

  1. We get the timestamp of when the message is received

  2. We create a RxMsg class with all the raw info from the wirepass.data input

  3. We parse the message payload, in order to extract the message type and additional arguments (if applicable)

  4. If the parsing is successful,

    • The message is displayed

    • If the mode == “request” then

      • We call the function on_message_temporal_node()

  5. If mode == “listen” - We just call the function on_message_main_node_listen()

Note

  1. It is the equivalent to the on_message_c() function of the mqtt_interaction_cloud class.

References

  1. It is useful to check the wirepas mqtt library documentation -> https://wirepas.github.io/wirepas-mqtt-library/

Parameters

data (wirepass.data) – a list containing various different information regarding the message. It includes the raw payload, the node id, the gateway id, the sink id.

publish_meshnet(gateway_id, sink_id, payload, node_id, prefix='')[source]

Publishes a message to the mesh network.

Does

  1. Tries to send the message to the mesh network using the wirepas_interaction_network function send_message(). For that to work, it has to give the following variables to the wirepas function:

    • gateway_id – that will transmit the message

    • sink_id – that will send the message to the right node

    • node_id – of the target node

    • src_ep

    • dst_ep

    • payload – of the message

    • Quality of Service (QOS) – of the desired MQTT connection

    • callback – function in case an error is reported

  2. Report in the logs if the publish was successful or not.

Note

It is the mesh-network equivalent to the publish_c() cloud interaction function.

Parameters

  • payload (bytes) – message payload, has a variable length

  • gateway_id (int) – gateway id to who the message was sent.

  • sink_id (int) – sink id that received the message in the gateway

  • node_id (int) – node id where we want to send the message.

  • prefix (string) – parent prefix to append to the current prefix. This will be reported in all prints made by this function

Returns

  • True if message was sent successfully, an error if not.

request_to_node(gw_id, sink_id, node_id, msg_type=None, args=[], payload=None)[source]

Sends messages to the mesh network.

Does
  1. If the payload is None:

    • It updates the message id, takes the last one from the local_settings.py file, and increments it by 1 by calling the function update_msg_id().

    • Puts all the message information to bytes using the message_to_bytes() function

  2. Stores the information regarding the message to be sent, in a database (if applicable)

  3. Gets the current timestamp in order to know when we sent the message, using the function get_timestamp()

  4. Sends the message by running the publish() function.

  5. Prints whether the message was sent or there was an error.

Note
This function can be called in 2 ways:

  1. providing the payload

  2. providing msg_type and arguments

In both cases you have to provide the gateway id, sink id, and node id.

Parameters

  • gw_id (int) – gateway id to who the message was sent.

  • sink_id (int) – sink id that received the message in the gateway

  • node_id (int) – node id where we want to send the message.

  • msg_type (int) – Type of message that we want to send

  • args (list) – Additional message arguments, depending on the message type

Returns

Nothing, so far

send_msg_to_nodes_from_input(prefix='')[source]

Prepares and sends a message to the nodes, based on the user input.

Does

  1. Update the message id, takes the last one from the local_settings.py file, and increments it by 1 by calling the function update_msg_id().

  2. Generates the initial message payload in bytes, by calling the function get_msg_from_input().

  3. It finally calls the request_to_node function and provides it the following parameters:

    • gw_id – gateway id

    • sink_id – sink id

    • dst_addr – id of the node we want to send the message to

    • payload – initial payload of the message in bytes, which so far just contains the message id and type.

Note

This is the mesh network equivalent function to the send_cloud_msg_from_input() function in the cloud class.
